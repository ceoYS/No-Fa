## 2. 문제 정의

### 2.1 사용자가 겪는 문제

1. 사용자는 의지가 없어서가 아니라, 반복되는 환경·시간대·감정·검색 경로에서 무너진다.
2. 단순 스트릭 앱은 기록은 해주지만, 실제 유혹 경로를 줄이지 못한다.
3. 차단 앱은 환경은 막아주지만, 감정·트리거·회복 기록을 쌓지 못한다.
4. 실패 후 수치심 때문에 앱을 삭제하거나 기록을 중단한다.
5. 한국어 기반 NoFap / 절제 앱은 문화적 맥락, 익명성, 수치심 완화, 책임 파트너 구조가 부족하다.
6. 민감한 주제이므로 “치료”, “죄”, “타락”, “실패자” 프레이밍을 쓰면 이탈과 브랜드 리스크가 커진다.

### 2.2 기존 앱의 한계

| 앱 유형 | 한계 |
|---|---|
| 단순 스트릭 앱 | 숫자 기록은 가능하지만 위기 순간의 행동 전환 장치가 약함 |
| 일반 습관 앱 | 성인 유해환경, 수치심, 익명 책임감, 실패 후 회복 UX를 다루지 않음 |
| 콘텐츠 차단 앱 | 차단 이후 사용자의 감정·트리거·회복 데이터를 쌓지 못함 |
| 커뮤니티 앱 | 비교, 조롱, 과도한 고백 문화로 수치심이 커질 수 있음 |
| AI 코칭 앱 | 민감 주제에서 부정확하거나 과도한 조언을 할 위험이 있음 |

## 7. MVP 범위

### 7.1 MVP와 차단 기능의 관계

MVP에는 “정식 차단 기능”을 포함하지 않는다.

MVP 단계에서는 제품 핵심 루프와 차단 기능의 기술 가능성, 권한 UX, 사용자 수요를 검증한다. 정식 차단 기능은 P0.5 Blocking Validation Gate를 통과한 뒤 P1에서 구현 여부를 결정한다.

| 단계 | 의미 |
|---|---|
| P0 | 제품 루프 검증 + 차단 기능 PoC |
| P0.5 | Blocking Validation Gate |
| P1 | 실제 차단 기능 정식화 여부 결정 및 구현 |
| P2 | 고급 차단, 프로 기능, AI 분석 등 확장 |

따라서 구현자는 MVP에서 실제 외부 사이트 차단 기능을 완성하려고 해서는 안 된다. MVP의 차단 관련 목표는 기술 가능성, 권한 요청 UX, 사용자 반응, 스토어 리스크를 검증하는 것이다.

## 8. Kakao Social Gate

한국 시장에서 책임 파트너는 단순 소셜 기능이 아니라 **사회적 락인 장치**다. 다만 카카오 친구 초대 구현은 기술·정책·개인정보 리스크가 있으므로 단계적으로 검증한다.

| 단계 | 정책 |
|---|---|
| P0 | 카카오 로그인은 선택. 이메일/Apple/Google 로그인으로 대체 가능 |
| P0 | 책임 파트너 “초대 의향”과 “공유 범위 설정 UI”를 검증 |
| P0 | 사용자가 어떤 정보를 공유하고 싶은지 조사 |
| P1 | 실제 카카오 친구 초대/파트너 연결 구현 검토 |
| 금지 | 연락처 전체 자동 수집 |
| 금지 | 친구에게 자동 알림 발송 |
| 금지 | 실패 기록 자동 전송 |
| 금지 | 파트너가 사용자를 감시하는 구조 |

### 8.1 P0 책임 파트너 검증 화면

P0에서는 실제 파트너 연결이나 카카오 친구 초대 기능을 구현하지 않더라도, 사용자의 공유 의향과 거부감을 검증하기 위한 프로토타입 화면은 포함한다.

| 화면 | 목적 | MVP 구현 수준 |
|---|---|---|
| 책임 파트너 소개 화면 | 혼자 버티지 않는 구조의 가치를 설명 | 프로토타입 포함 |
| 공유 범위 선택 화면 | 사용자가 어떤 정보를 공유할 수 있는지 검증 | 프로토타입 포함 |
| 초대 전 확인 화면 | 실제 초대 직전 불안 요소 확인 | 프로토타입 포함 |
| 나중에 하기 화면 | 소셜 기능 거부감 측정 | 프로토타입 포함 |

### 8.2 검증 목적

1. 사용자가 책임 파트너 구조를 가치 있게 느끼는지 확인한다.
2. 어떤 정보 공유를 허용하는지 확인한다.
3. 실패 기록 공유에 대한 거부감을 확인한다.
4. 카카오 초대 구현 전 소셜 락인 니즈를 검증한다.
5. 책임 파트너가 “감시자”가 아니라 “동의 기반 응원자”로 인식되는지 확인한다.

## 11. 캐릭터 성장 시스템 PRD

### 11.1 캐릭터 역할

캐릭터는 사용자를 평가하는 존재가 아니라, 사용자의 자기통제 여정을 시각화하는 동료다.

### 11.2 세계관 옵션

| 옵션 | 컨셉 | 장점 | 단점 | 적합도 |
|---|---|---|---|---|
| Option A | 씨앗/식물형 | 성장, 회복, 루틴과 잘 맞음 | 흔하고 밋밋할 수 있음 | 중 |
| Option B | 수호자/동료형 | 차단 기능과 잘 맞음 | 과도한 판타지화 위험 | 상 |
| Option C | 방/공간 성장형 | 꾸미기·아이템과 자연스럽게 연결 | 구현 범위 커짐 | 중상 |

### 11.3 추천안

`추천`: **Option B + Option C 혼합**

| 요소 | 방향 |
|---|---|
| 캐릭터 | 절제 환경을 함께 지키는 수호자/동료 |
| 홈 공간 | 사용자의 절제 루틴이 쌓이는 “내 절제 공간” |
| 성장 방식 | 스트릭, 체크인, 위기 극복, 회복 플랜으로 성장 |
| 꾸미기 | 캐릭터 외형, 공간 테마, 오브젝트 |
| 금지 | 전투력, 능력치, 캐릭터 사망, 실패 처벌, 유료 성장 부스팅 |

### 11.4 성장 연결

| 사용자 행동 | 캐릭터 반응 | 보상 |

### 15.2 Mobbin Missing App Rule

- 후보 앱이 Mobbin에 없으면 억지로 찾지 않는다.
- 같은 Collection 안에서 같은 UX 패턴을 가진 대체 앱 2~3개를 찾는다.
- 앱 기준이 아니라 화면 패턴 기준으로 저장한다.
- Collection 이름을 먼저 정하고, 화면은 해당 Collection에만 넣는다.
- 각 Collection은 최소 5개, 최대 20개 화면까지만 수집한다.
- 중복 화면, 단순 스플래시, 광고성 Paywall 반복 화면은 제외한다.
- Mobbin은 UI 복제용이 아니라 패턴 분류용이다.

## 28. PRD Lock 이후 문서 분리 원칙

PRD는 제품 목적, 범위, 우선순위의 최상위 source of truth다. PRD Lock 이후에는 PRD를 계속 비대하게 확장하지 않고, 하위 문서로 분리해 관리한다.

| 문서 | 역할 | PRD와의 관계 |
|---|---|---|
| PRD.md | 제품 목적, 범위, 우선순위, 핵심 원칙 | 최상위 source of truth |
| UX_RULES.md | UX 톤, 실패/회복 플로우, 차단 화면 태도, 책임 파트너 UX 판단 기준 | PRD의 UX 원칙을 상세화 |
| SCREEN_INVENTORY.md | MVP 화면 목록, 화면별 목적, CTA, 금지 카피, 검수 포인트 | PRD의 화면 목록을 구현 전 수준으로 분해 |
| COPY_POLICY.md | 권장/금지 카피, 민감 표현, 앱스토어 표현, 차단/실패/결제 문구 정책 | PRD의 카피 원칙을 운영 가능하게 분리 |
| BLOCKING_TECH_POC_PLAN.md | iOS/Android 차단 기술 검증, 권한 UX, 스토어 심사 리스크, 로그 최소화 | PRD의 차단 PoC Gate를 실행 문서화 |
| CHARACTER_SYSTEM_SPEC.md | 캐릭터 세계관, 성장 단계, 보상, 실패 처리, 꾸미기 원칙 | PRD의 캐릭터 성장 시스템을 상세화 |
| MONETIZATION_POLICY.md | 무료/프로/단품 결제/뽑기 금지선/미성년자 보호/과소비 방지 | PRD의 수익화 원칙을 정책화 |
| MOBBIN_BENCHMARK_PLAN.md | Collection 구조, 앱 후보, Missing App Rule, 수집 기준 | PRD의 Mobbin 계획을 실행 가능하게 분리 |
| DESIGN_DIRECTION.md | 시각 톤, 브랜드 무드, 캐릭터 방향, UI 분위기 | PRD와 UX_RULES를 기반으로 작성 |

### 28.1 하위 문서 작성 원칙

- PRD와 충돌하는 내용은 하위 문서에 넣지 않는다.
- 하위 문서에서 새로운 기능을 추가하지 않는다.
- 새로운 기능 아이디어는 PRD 변경 요청으로 먼저 올린다.
- Claude Code 또는 디자인 도구에 입력할 때는 전체 PRD보다 목적에 맞는 하위 문서를 우선 사용한다.
- 구현 판단은 PRD.md, UX 판단은 UX_RULES.md, 화면 판단은 SCREEN_INVENTORY.md, 카피 판단은 COPY_POLICY.md를 우선한다.
