# BLOCKING_TECH_POC_PLAN.md

차단 기능 기술 가능성 검증 계획 (PoC 선행)

상위 문서: PRD §7.3, §9, §10, §26 / 같은 라운드: UX_RULES.md, COPY_POLICY.md
이 문서는 **계획 문서**다. 차단 기능을 구현하지 않는다.

> 중요 금지: 차단 구현 금지 · 성인 사이트명 나열 금지 · 성인 URL/검색어 저장 금지 · 완벽 차단 주장 금지 · 실제 차단 목록 작성 금지 · 네트워크 호출 금지. 모든 차단 동작은 **가능성 계획**으로만 다룬다.

---

## 1. 목적 (Purpose)

우리가 만들고 싶은 차단이 아니라, **스토어 심사와 OS 제약 안에서 실제 가능한 차단**을 확인한다 (PRD §9.1).
정식 차단(REQ-007B)은 P0.5 Gate 통과 후 P1에서 구현 여부를 결정한다.

---

## 2. Non-goals

- 실제 차단 기능 구현
- 실제/기본 차단 목록(블록리스트) 작성
- 성인 사이트명·URL·검색어 수집/저장/노출
- 네트워크 호출, 패키지 설치
- "완벽/절대" 차단 보장

---

## 3. 왜 차단이 PoC 먼저인가 (Why blocking is PoC first)

- 차단은 OS/스토어 정책에 크게 의존하며 구현 리스크가 높다 (PRD §26: 가능성 높음/영향 높음).
- 우회 가능성이 본질적으로 존재하므로 "완벽 차단"을 약속할 수 없다.
- 성인 콘텐츠 제공 앱으로 오해될 심사 리스크가 있다.
- 차단 로그가 민감정보가 될 수 있어 데이터 설계가 선행돼야 한다.
- 따라서 "가능한 범위"를 먼저 확정하고, 그 위에서만 정식 구현을 판단한다.

---

## 4. iOS 가능성 질문 (iOS feasibility questions)

- 의도한 차단 범위(앱 단위 / 도메인 단위 / 브라우저 단위)를 iOS가 지원하는가?
- Screen Time / Managed Settings / content filter 계열 중 어떤 경로가 정책상 허용되는가?
- 어떤 권한·자격(entitlement)이 필요한가? 일반 앱 배포에서 가능한가?
- 사용자가 직접 OS 설정을 켜는 가이드 방식이 더 현실적인가?
- 산출물: iOS 차단 가능성 매트릭스 (범위 × 가능 여부 × 필요 권한 × 제약).

---

## 5. Android 가능성 질문 (Android feasibility questions)

- VPN / DNS / Accessibility / Device Policy 중 의도한 차단에 적합하고 **안전한** 방식은?
- 위험·고지 권한(예: Accessibility 오남용 우려) 없이 가능한 경로가 있는가?
- Google Play 정책상 어떤 권한 사용이 심사 리스크를 키우는가?
- 산출물: Android 차단 가능성 매트릭스 (방식 × 가능 여부 × 권한 위험 × 정책 리스크).

---

## 6. 권한 UX 질문 (Permission UX questions)

- 사용자가 권한 요청 **이유**를 이해하는가? (이유 먼저 설명 — UX_RULES §5)
- 권한 요청이 강제·압박처럼 느껴지지 않는가?
- 권한 거부 시 대안 경로(§11)를 명확히 안내하는가?
- 차단 해제 딜레이가 벌이 아닌 **자기 설정**으로 인지되는가?

---

## 7. 스토어 심사 리스크 (Store review risks)

- 성인 콘텐츠 제공/접근 앱으로 오해될 표현·기능이 없는가?
- 앱 내 성인 콘텐츠/링크/사이트명 0 (PRD §10.3, §25).
- 사용하는 권한이 스토어 정책 위반·고위험으로 분류되지 않는가?
- "완벽 차단" 등 보장성 문구가 심사 설명/메타데이터에 없는가?
- 산출물: 심사 체크리스트.

---

## 8. 개인정보/로깅 리스크 (Privacy/logging risks)

PRD §17, §18 기준.
- 차단 이벤트를 **원문(raw URL/검색어) 없이** 측정 가능한가?
- 차단 목록은 암호화·삭제 가능 설계인가?
- 금지 속성: `raw_url`, `raw_search_query`, `adult_site_name`, 민감 이벤트 `exact_timestamp`.
- 허용 측정: `blocking_level`(weak/standard/strong), 차단 이벤트 count, time_of_day bucket.

---

## 9. 차단 카피 리스크 (Blocking copy risks)

COPY_POLICY §7 기준.
- "완벽 차단/절대 우회 불가" 금지.
- "해제하면 당신 책임" 류 위협 금지.
- 차단 화면 "부끄럽지 않나요" 류 꾸짖기 금지 → 멈춤 + 대체 활동.
- 산출물: 차단 관련 금지어 목록.

---

## 10. PoC 성공 기준 (PoC success criteria)

PRD §7.3 기준. 다음을 모두 만족하면 통과.
- iOS·Android 중 **최소 1개** 실사용 가능한 차단 경로 식별.
- 권한 UX가 강압적으로 느껴지지 않음 (설명 후 과도한 이탈 없음).
- 앱 내 성인 콘텐츠 저장/노출/링크 0.
- raw URL 저장 없이 핵심 지표 측정 가능.
- 앱스토어/구글플레이 리스크가 문서화됨.
- 정식 차단이 불가능할 경우의 대안 경로(§11) 존재.

---

## 11. PoC 실패 시 대안 (PoC failure alternatives)

PRD §9.3 기준.
- OS 기본 차단 설정 가이드 (사용자가 직접 켜기).
- 브라우저 사용 축소 체크리스트 (특정 시간대 사용 감소).
- 긴급 차단 Lite (앱 내 위기 버튼·대체 활동·알림만).
- 비브랜드 외부 차단 가이드 (특정 앱 홍보 없이 일반 설정 안내).

---

## 12. 사람 검수 체크리스트 (Human review checklist)

- [ ] 차단을 구현하지 않고 가능성만 다뤘는가?
- [ ] 성인 사이트명/URL/검색어/블록리스트가 어디에도 없는가?
- [ ] "완벽 차단" 보장 표현이 없는가?
- [ ] iOS/Android 가능성 매트릭스 산출 계획이 있는가?
- [ ] raw URL 없이 지표 측정하는 설계인가?
- [ ] 권한 UX가 강압적이지 않은가?
- [ ] 스토어 심사 체크리스트가 있는가?
- [ ] 실패 시 대안 경로가 정의됐는가?
