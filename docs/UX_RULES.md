# UX_RULES.md

Korean NoFap / 절제 / 자기통제 앱 — UX 규칙

상위 문서: PRD §5, §13, §16 / 같은 라운드: COPY_POLICY.md, SCREEN_INVENTORY.md
이 문서는 PRD의 UX 원칙을 상세화한 판단 기준이다. 화면 구현·디자인 생성은 하지 않는다.

---

## 1. UX 미션 (UX mission)

사용자가 혼자 버티지 않도록, **죄책감이 아닌 회복**으로 다시 돌아오게 만든다.
민감 주제이므로 모든 UX 결정은 "이 화면이 사용자를 깎아내리는가, 다시 일어서게 하는가"를 먼저 묻는다.

---

## 2. 제품 톤 (Product tone)

- 담담하고 비판 없음. 과장·훈계·종교적 어조 금지.
- 가볍지만 유치하지 않음. 진지하지만 무겁지 않음.
- 위기 순간엔 짧고 단호하게. 회복 순간엔 따뜻하게.
- "내가 정한 환경을 지킨다"는 사용자 주도 프레임을 유지한다.

---

## 3. 회복 우선 원칙 (Recovery-first principle)

PRD §5.3 기준.

- 실패는 끝이 아니라 회복 루프의 시작점이다.
- 모든 실패/리셋 화면은 24h 재방문을 유도하도록 설계한다.
- "다시 시작"은 처벌 후 복귀가 아니라, 패턴 학습의 일부로 제시한다.
- 스트릭이 끊겨도 누적 데이터·캐릭터·여정은 사라지지 않음을 보여준다.

---

## 4. 실패/리셋 UX 규칙 (Failure/relapse UX rules)

| Do | Don't |
|---|---|
| 원인·감정·다음 행동을 차분히 기록 | 실패자/중독자 프레이밍 |
| 회복 플랜으로 즉시 연결 | 벌점·강등·캐릭터 처벌 |
| 비공개 기본, 공유는 사용자 선택 | 실패 자동 공유/파트너 자동 알림 |
| "여정이 끝난 건 아니에요" 톤 | "또 실패했네요" 톤 |
| 기록은 1분 이내로 가볍게 | 길고 추궁하는 질문 |

성적 고백을 강요하지 않는다. 트리거/감정은 enum 선택 중심, 자유서술은 선택·비공개.

---

## 5. 차단 UX 규칙 (Blocking UX rules)

PRD §9, §10 / BLOCKING_TECH_POC_PLAN.md 기준. **MVP는 PoC만. 실제 차단 구현 금지.**

| Do | Don't |
|---|---|
| 사용자가 직접 차단 켜기 선택 | 권한 필수·강제 압박 |
| 권한 요청 이유를 먼저 설명 | 이유 없는 권한 요청 |
| 차단=내가 정한 환경 지키는 장치 | 차단=강제로 막아주는 장치 |
| 해제 딜레이를 자기 설정으로 설명 | 딜레이를 벌/통제로 표현 |
| 차단 도달 시 "멈춰 세우기" | 꾸짖기/부끄럽게 하기 |
| OS/스토어 한계를 정직하게 고지 | "완벽 차단/절대 우회 불가" 보장 |

차단 화면은 비난하지 않고 **멈춤 → 대체 활동**으로 전환한다.

---

## 6. 위기 버튼 UX 규칙 (Crisis button UX rules)

- 1탭 진입. 진입 장벽·확인 단계 최소화.
- 큰 버튼, 짧은 카피, 타이머/대체 활동으로 즉시 연결.
- "참아야 한다"식 명령 금지. "지금은 선택을 늦추는 시간" 톤.
- 위기 지원(대체 활동, 멈춤)은 무료 핵심 기능. **유료화 금지** (MONETIZATION_POLICY).
- 위기 버튼 사용은 성취로 인정한다(캐릭터 XP). 사용했다고 깎지 않는다.

---

## 7. 캐릭터 UX 규칙 (Character UX rules)

CHARACTER_SYSTEM_SPEC.md 기준.

- 캐릭터는 평가자가 아니라 동료/수호자. **절대 사용자를 비난하지 않는다.**
- 실패 시 캐릭터는 "쉬는 상태"이지 죽거나 우는 상태가 아니다.
- 성장은 행동(스트릭·체크인·위기 극복·회복 플랜)의 결과로만.
- 유료 성장 부스팅·스트릭 복구·실패 삭제로 캐릭터를 키우지 않는다(pay-to-win 금지).

---

## 8. 책임 파트너 UX 규칙 (Responsibility partner UX rules)

PRD §8 기준. **P0는 검증 프로토타입만. 실제 카카오 초대 구현 금지.**

| Do | Don't |
|---|---|
| 공유 범위를 사용자가 직접 선택 | 공유 범위 기본 ON |
| 동의 기반 응원자 프레임 | 감시자 프레임 |
| 실패 여부/원인은 기본 공유 금지 | 실패 기록 자동 전송 |
| 초대 전 확인·나중에 하기 제공 | 자동 친구 초대·연락처 전체 수집 |
| 위기는 사용자 선택 시에만 요약 공유 | 위기 상황 자동 알림 |

---

## 9. 코호트/리더보드 UX 규칙 (Cohort/leaderboard UX rules)

PRD §25 (공개 망신형 리더보드 금지) 기준. **P1 기능.**

- 익명 ID 중심. 실명·개인 식별 금지.
- 비교 최소화. "꼴찌/뒤처짐" 표현 금지.
- 그룹 목표·함께 버팀 중심. 순위 경쟁보다 공동 진척.

---

## 10. 개인정보·동의 UX 규칙 (Privacy and consent UX rules)

PRD §17, §18 기준.

- 항목별 동의. 자동 공유 금지. 기본값 비공개.
- 민감 데이터(체크인·트리거·감정·차단)는 기본 비공개.
- 차단 로그·검색어·URL 원문 저장 금지. 측정은 enum/bucket/count로.
- 동의 철회·삭제 경로를 명확히 제공.

---

## 11. Paywall UX 규칙 (Paywall UX rules)

MONETIZATION_POLICY.md 기준. **P2 기능.**

| Do | Don't |
|---|---|
| 핵심 절제 루프는 무료 유지 | "무료는 부족" 압박 |
| 프로=더 세밀한 분석 선택 | 위기 지원·회복을 유료화 |
| 기능 비교를 투명하게 | 한 번 더 뽑기/과금 압박 |
| 비랜덤 단품/구독만 | 스트릭 복구·실패 삭제·차단 해제권 판매 |

---

## 12. 사람 검수 체크리스트 (Human review checklist)

화면/카피/플로우가 다음 중 하나라도 해당하면 **사람 검수 필요(Needs human review)**:

- [ ] 사용자를 비난·수치심 유발할 여지가 있는가?
- [ ] 종교적·도덕적 판단을 암시하는가?
- [ ] 치료/완치 효과를 단정하는가?
- [ ] "완벽 차단/절대 우회 불가"를 약속하는가?
- [ ] 실패를 자동 공유하거나 파트너를 감시자로 만드는가?
- [ ] 공유 범위가 사용자 선택 없이 기본 ON인가?
- [ ] 위기 지원/회복을 유료 뒤에 두는가?
- [ ] 성인 콘텐츠/사이트명/URL을 노출·저장하는가?
- [ ] 권한 요청이 강제·압박처럼 느껴지는가?
- [ ] 캐릭터가 사용자를 평가·처벌하는가?

### 위험 UX 패턴 예시 (Examples of risky UX patterns)

- 온보딩에서 "얼마나 심각한가요?" 류 자기 진단 강요
- 실패 시 빨간 경고·하강 애니메이션으로 죄책감 자극
- 차단 화면에서 "부끄럽지 않나요?" 류 훈계
- 리더보드 최하위 강조
- 권한 거부 시 앱 사용 불가로 압박
- 결제 후 "실패 기록 삭제" 제안
