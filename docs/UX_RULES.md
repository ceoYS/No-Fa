# UX_RULES.md

Korean NoFap / 절제 / 자기통제 앱 — UX 규칙

상위 문서: PRD §5, §13, §16 / 같은 라운드: COPY_POLICY.md, SCREEN_INVENTORY.md
이 문서는 PRD의 UX 원칙을 상세화한 판단 기준이다. 화면 구현·디자인 생성은 하지 않는다.

---

## 1. UX 미션 (UX mission)

사용자가 혼자 버티지 않도록, **죄책감이 아닌 회복**으로 다시 돌아오게 만든다.
민감 주제이므로 모든 UX 결정은 "이 화면이 사용자를 깎아내리는가, 다시 일어서게 하는가"를 먼저 묻는다.

---

## 2. 제품 톤 (Product tone)

- 담담하고 비판 없음. 과장·훈계·종교적 어조 금지.
- 가볍지만 유치하지 않음. 진지하지만 무겁지 않음.
- 위기 순간엔 짧고 단호하게. 회복 순간엔 따뜻하게.
- "내가 정한 환경을 지킨다"는 사용자 주도 프레임을 유지한다.

---

## 3. 회복 우선 원칙 (Recovery-first principle)

PRD §5.3 기준.

- 실패는 끝이 아니라 회복 루프의 시작점이다.
- 모든 실패/리셋 화면은 24h 재방문을 유도하도록 설계한다.
- "다시 시작"은 처벌 후 복귀가 아니라, 패턴 학습의 일부로 제시한다.
- 스트릭이 끊겨도 누적 데이터·캐릭터·여정은 사라지지 않음을 보여준다.

---

## 4. 실패/리셋 UX 규칙 (Failure/relapse UX rules)

| Do | Don't |
|---|---|
| 원인·감정·다음 행동을 차분히 기록 | 실패자/중독자 프레이밍 |
| 회복 플랜으로 즉시 연결 | 벌점·강등·캐릭터 처벌 |
| 비공개 기본, 공유는 사용자 선택 | 실패 자동 공유/파트너 자동 알림 |
| "여정이 끝난 건 아니에요" 톤 | "또 실패했네요" 톤 |
| 기록은 1분 이내로 가볍게 | 길고 추궁하는 질문 |

성적 고백을 강요하지 않는다. 트리거/감정은 enum 선택 중심, 자유서술은 선택·비공개.

---

## 5. 차단 UX 규칙 (Blocking UX rules)

PRD §9, §10 / BLOCKING_TECH_POC_PLAN.md 기준. **MVP는 PoC만. 실제 차단 구현 금지.**

| Do | Don't |
|---|---|
| 사용자가 직접 차단 켜기 선택 | 권한 필수·강제 압박 |
| 권한 요청 이유를 먼저 설명 | 이유 없는 권한 요청 |
| 차단=내가 정한 환경 지키는 장치 | 차단=강제로 막아주는 장치 |
| 해제 딜레이를 자기 설정으로 설명 | 딜레이를 벌/통제로 표현 |
| 차단 도달 시 "멈춰 세우기" | 꾸짖기/부끄럽게 하기 |
| OS/스토어 한계를 정직하게 고지 | "완벽 차단/절대 우회 불가" 보장 |

차단 화면은 비난하지 않고 **멈춤 → 대체 활동**으로 전환한다.

## 6. 위기 버튼 UX 규칙 (Crisis button UX rules)

- 1탭 진입. 진입 장벽·확인 단계 최소화.
- 큰 버튼, 짧은 카피, 타이머/대체 활동으로 즉시 연결.
- "참아야 한다"식 명령 금지. "지금은 선택을 늦추는 시간" 톤.
- 위기 지원(대체 활동, 멈춤)은 무료 핵심 기능. **유료화 금지** (MONETIZATION_POLICY).
- 위기 버튼 사용은 성취로 인정한다(캐릭터 XP). 사용했다고 깎지 않는다.

## 7. 캐릭터 UX 규칙 (Character UX rules)

CHARACTER_SYSTEM_SPEC.md 기준.

- 캐릭터는 평가자가 아니라 동료/수호자. **절대 사용자를 비난하지 않는다.**
- 실패 시 캐릭터는 "쉬는 상태"이지 죽거나 우는 상태가 아니다.
- 성장은 행동(스트릭·체크인·위기 극복·회복 플랜)의 결과로만.
- 유료 성장 부스팅·스트릭 복구·실패 삭제로 캐릭터를 키우지 않는다(pay-to-win 금지).

## 8. 책임 파트너 UX 규칙 (Responsibility partner UX rules)

PRD §8 기준. **P0는 검증 프로토타입만. 실제 카카오 초대 구현 금지.**

| Do | Don't |
|---|---|
| 공유 범위를 사용자가 직접 선택 | 공유 범위 기본 ON |
| 동의 기반 응원자 프레임 | 감시자 프레임 |
| 실패 여부/원인은 기본 공유 금지 | 실패 기록 자동 전송 |
| 초대 전 확인·나중에 하기 제공 | 자동 친구 초대·연락처 전체 수집 |
| 위기는 사용자 선택 시에만 요약 공유 | 위기 상황 자동 알림 |

## 10. 개인정보·동의 UX 규칙 (Privacy and consent UX rules)

PRD §17, §18 기준.

- 항목별 동의. 자동 공유 금지. 기본값 비공개.
- 민감 데이터(체크인·트리거·감정·차단)는 기본 비공개.
- 차단 로그·검색어·URL 원문 저장 금지. 측정은 enum/bucket/count로.
- 동의 철회·삭제 경로를 명확히 제공.

### 10.1 비공개 표기 UI 규칙 (Privacy label UI rules)

기본 원칙: **NoF의 모든 사용자 기록은 기본 비공개다.** 사용자가 "공유하기/게시하기/내보내기/파트너에게 보내기" 같은 명시적 행동을 하지 않는 한 어떤 기록도 외부로 나가지 않는다.

| Do | Don't |
|---|---|
| 비공개는 기본값으로 가정한다 | "나만 보는 기록", "비공개 기록"을 홈/상세의 반복 배지로 노출 |
| 비공개 안내는 온보딩 · 설정 · 도움말에 배치 | 모든 메트릭 카드마다 비공개 라벨 부착 |
| 공유/내보내기 직전 한 줄 재확인 표시 | 평소 화면에서 정책 문구로 사용자 안심시키기 |
| 체크인 등 입력형 화면에 한 문장만 안내 | 같은 안내 문구를 여러 화면에서 반복 |
| 미래 공유 행동은 사전 미리보기 + 명시 동의 | 자동/암묵 공유 또는 옵트아웃 기본 ON |

권장 카피:

- 체크인 단일 안내(필요 시 1문장만): "공유하지 않는 한 기록은 밖으로 나가지 않아요."
- 공유 직전 안내: "이 내용을 공유할까요?" / "공유 전 내용을 확인하세요."

P0 범위: 공개 피드/게시/공유 기능은 **포함하지 않는다.** 미래의 공유/내보내기/파트너 전송은 명시적·동의 기반·미리보기 기반으로만 만든다.

## 10.5 나의 규율 UX 규칙 (My Discipline UX rules)

PRD §0.5.9·§0.5.9.1·§0.5.9.2 기준. 규율은 사용자가 직접 정한 사적 자기통제 기준이다. **P0는 상태 표시 + 행별 부분 갱신 + 가벼운 로컬 추가/삭제 + 체크인 통합 점검까지. 본격 편집 UI 금지.**

| Do | Don't |
|---|---|
| 비공개 기본·자기 정의 | 외부/파트너 자동 공유 |
| 5-state 상태 칩(지켰어요 / 확인 필요 / 흔들렸어요 / 회복 완료 / 오늘은 해당 없음) | "위반"·"벌점"·"실패"·"포기" 1차 카피 |
| "내가 정한 기준" 프레이밍 | "약속/맹세/공약" 공개 약속 프레이밍 |
| 단호하되 권위주의적이지 않은 톤 | 명령형 ("지켜야 한다") |
| `흔들렸어요` → 같은 카드 안에 회복 행동 제안 + 회복 화면 보조 라우팅 | `흔들렸어요`를 처벌·강등·잠금으로 연결 |
| 회복 행동 완료 → 자동 `회복 완료` 전이 | 회복 후에도 `흔들렸어요` 상태 고착 |
| 규율을 점수·랭킹 없이 표시 | 규율을 점수/등급/랭킹으로 환산 |
| P0 가벼운 로컬 추가/삭제·상태 갱신 허용 | 추가/삭제를 "위반/포기/실패" 카피로 묘사 |
| 삭제 확인 카피는 회복 우선("이 규율을 삭제할까요?") | "이 규율을 포기할까요?" 류 |

### 10.5.1 Discipline 확인 입력 모달리티 (Discipline confirmation input)

PRD §0.5.9.2 — 규율 상태는 P0에서 **사용자 수동 확인**으로만 갱신된다(자동 판정·자동 시간 리셋 없음).

| Do | Don't |
|---|---|
| 체크인 마지막 스텝에 "오늘의 규율 점검" 카드 포함 (`지켰어요` / `흔들렸어요` / `해당 없음` 3-tap 칩) | 체크인을 끝낼 때 모든 규율을 자동으로 `지켰어요`로 처리 |
| 디시플린 화면에서 행별 5-state 변경 시트 제공 | 강제 일괄 갱신만 허용 |
| 미선택 규율은 `확인 필요` 유지 (비난 어휘 아님) | 미선택을 `흔들렸어요`로 자동 분류 |
| `흔들렸어요` 선택 시 회복 행동 1개 제안 한 줄 + 회복 화면 보조 CTA | `흔들렸어요` 직후 자동 라우팅으로 사용자 컨트롤 박탈 |

규율 UI는 사용자가 "오늘 내가 정한 기준을 지키고 있는가"를 비공개로 점검하는 도구다. 결코 벌주거나 비난하지 않는다. **P0에서 영속화·본격 편집 UI·템플릿·위험 시간대 리마인더·규율별 인사이트는 다루지 않는다(PRD §0.5.9).**

## 10.6 Discipline 정량 모델 UX 규칙 (Discipline quantitative model UX rules)

PRD §0.5.10 기준. 여러 규율의 상태를 내부 정량 신호로 모아 펫·방의 표현 상태에 부드럽게 반영하되, **사용자에게 점수·등급·랭킹으로 노출하지 않는다.**

| Do | Don't |
|---|---|
| 내부 일관성(consistency) / 방 온기(warmth)는 수치 계산만 | 점수·퍼센트·등급 노출 ("72점", "78%") |
| `weight`는 "나에게 더 중요한 기준" 표시 | `weight`를 가중 점수·랭킹 환산 노출 |
| 방 온기는 상태 어휘로 표시 (약함/잔잔함/안정/따뜻함) | 빨강·경고·하강 시각 톤 |
| `needs_check` 누적 = 회복 루틴 제안 | `needs_check` 누적 = 펫 강등·잠금·차단 |
| 상태 변화는 잔불·조명·온도의 연속 스펙트럼 | 이진 성공/실패 표시 |
| 정량 계산값은 비공개·로컬 | 정량 계산값을 외부/파트너에 자동 공유 |
| 가중치는 사용자 자기 정의 | 시스템이 임의로 "중요도" 평가 |

규칙: 어떤 입력도 펫을 해치지 않으며, 어떤 출력도 사용자를 평가하지 않는다. 정량 모델은 펫의 방이 얼마나 따뜻한지를 한 가지 부드러운 신호로 모으는 도구이지, 사용자를 점수화하는 도구가 아니다(§0.5.10 F 상속).

## 10.7 Recovery Calendar UX 규칙 (Recovery Calendar UX rules)

PRD §0.5.3·§0.5.3.1·§0.5.3.2 기준. P0의 캘린더는 **최근 7일을 1차 표면으로** 보여주며, 가짜 월 placeholder를 두지 않는다.

| Do | Don't |
|---|---|
| 최근 7일 점은 모두 **탭 가능**, 탭 시 그 날의 한 줄 요약 표시 | 점을 장식으로만 두기 (탭 불가) |
| 한 줄 요약은 enum 신호만 (체크인 여부 + 흔들림 수 + 회복 행동 수) | 자유서술·사진 첨부·민감 맥락 포함 |
| 사용자 라벨은 직접 어휘 (`지킨 날` / `회복한 날` / `확인 필요` / `기록 전`) | 메타포 단독 라벨 (`이어온 날`, `조용해진 날`, `아직`) |
| 화면 1차 타이틀은 `최근 기록` 또는 `절제 기록` | 화면 타이틀로 `잔불 캘린더` 메타포 단독 사용 |
| 톤은 잔불·조명·온도의 연속 스펙트럼 | 빨강·이진(성공/실패) 톤 |
| 월/주 전체 보기는 도움말·설정 한 줄 안내까지만 (P1 게이트) | 사용자 UI에 "월/주 전체 보기는 P1에서 열려요" 류 단계 placeholder 노출 |
| 점수·퍼센트·등급 비표시 | 점수·랭킹·"브레인 리와이어링 %" 류 |

## 10.8 미래 기능 placeholder 규칙 (Future-feature placeholder rules)

PRD §0.5.8·§0.5.8.1 기준. **단계 어휘는 운영 어휘다. 사용자 UI 카피로 노출하지 않는다.**

| Do | Don't |
|---|---|
| 미래 기능이 P0에 없으면 placeholder를 만들지 않는다 | "월/주 전체 보기는 P1에서 열려요" 류 placeholder 카피 노출 |
| 안내가 필요하면 도움말·설정에 한 줄로만 둔다 | 핵심 화면에 "다음 단계 (P1)" / "프로토타입" / "보기 전용" 라벨 노출 |
| 일반 어휘로 톤을 맞춘다 ("곧 열려요") | 단계 어휘(`P0`, `P0.1`, `P0.5`, `P1`, `P2`, `MVP`, `프로토타입`, `보기 전용`, `데모`, `mock`) 노출 |
| 운영 어휘(`enum`, `bucket`, `정책`, `코스메틱`, `능력치`)는 운영 문서에만 | 사용자 화면에 `enum`/`정책` 등 운영 어휘 노출 |

미래 기능을 사용자에게 약속하지 않는다 — 구체 시점 약속도 피한다("다음 분기에", "한 달 안에" 류 회피).

## 11. Paywall UX 규칙 (Paywall UX rules)

MONETIZATION_POLICY.md 기준. **P2 기능.**

| Do | Don't |
|---|---|
| 핵심 절제 루프는 무료 유지 | "무료는 부족" 압박 |
| 프로=더 세밀한 분석 선택 | 위기 지원·회복을 유료화 |
| 기능 비교를 투명하게 | 한 번 더 뽑기/과금 압박 |
| 비랜덤 단품/구독만 | 스트릭 복구·실패 삭제·차단 해제권 판매 |

## 12. 사람 검수 체크리스트 (Human review checklist)

화면/카피/플로우가 다음 중 하나라도 해당하면 **사람 검수 필요(Needs human review)**:

- [ ] 사용자를 비난·수치심 유발할 여지가 있는가?
- [ ] 종교적·도덕적 판단을 암시하는가?
- [ ] 치료/완치 효과를 단정하는가?
- [ ] "완벽 차단/절대 우회 불가"를 약속하는가?
- [ ] 실패를 자동 공유하거나 파트너를 감시자로 만드는가?
- [ ] 공유 범위가 사용자 선택 없이 기본 ON인가?
- [ ] 위기 지원/회복을 유료 뒤에 두는가?
- [ ] 성인 콘텐츠/사이트명/URL을 노출·저장하는가?
- [ ] 권한 요청이 강제·압박처럼 느껴지는가?
- [ ] 캐릭터가 사용자를 평가·처벌하는가?

### 위험 UX 패턴 예시 (Examples of risky UX patterns)

- 온보딩에서 "얼마나 심각한가요?" 류 자기 진단 강요
- 실패 시 빨간 경고·하강 애니메이션으로 죄책감 자극
- 차단 화면에서 "부끄럽지 않나요?" 류 훈계
- 리더보드 최하위 강조
- 권한 거부 시 앱 사용 불가로 압박
- 결제 후 "실패 기록 삭제" 제안
